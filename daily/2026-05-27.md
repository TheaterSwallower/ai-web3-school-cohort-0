## 学习笔记

### 关键概念

**Agent Trust & Reputation 核心思想**：信任不是一个分数，而是一组可追溯、可比较、可解释的证据。Agent 的可信度来自可验证行为，而不是自我声明。

**七个知识节点**：

| 节点 | 难度 | 一句话 |
|------|------|--------|
| Reputation | 初级 | 历史表现信号集合：成功率/响应时间/争议率/退款率 — 但要按任务类型拆开，不能压成黑盒分数 |
| Review | 初级 | 绑定任务 ID/交付物/付款记录的评价 — 一个含具体问题的 review 比十个 "good job" 更有价值 |
| Attestation | 中级 | 某主体对 Agent/任务/结果的可验证声明：issuer → subject → claim → evidence → expiration → revocation |
| Stake | 中级 | 经济抵押让承诺有成本 — 但 stake 多≠能力强，小团队可能 stake 少但质量高 |
| Slashing | 高级 | 违约时罚没抵押 — 必须明确：什么算违约 / 谁来仲裁 / 误罚怎么处理 / 申诉机制 |
| Validation | 中级 | 能力验证 vs 任务结果验证 — 最好做成可查询历史，而不是贴个认证徽章 |
| ERC-8004 | 高级 | Agent 身份/声誉/验证的标准 registry — 不试图做单一评分，而是提供公共承载层 |

**三条核心原则**：
1. **声誉绑定身份**：没有稳定身份（上一章），历史记录无法积累
2. **评价绑定任务**：泛泛五星评价不如具体任务结果 + 交付 hash
3. **惩罚要真实**：没有成本的承诺容易滥用 — stake/slashing 让失信有代价

**本章在知识地图的位置**：Agent Trust & Reputation 是 Agent Identity（Day 6）的直接延续，也是 Machine Payment（Day 5）的信任基础。没有信任层，Agent 之间无法安全委托；没有声誉和验证，用户难以判断哪个 Agent 值得付款。

### 设计启示

声誉系统常见陷阱：
- ❌ 把所有信号压成一个分数 → 用户看不到适用场景差异
- ❌ 评价不绑定任务 → 变成可刷的口碑文本
- ❌ 只有好评没有争议机制 → 刷评无法发现
- ❌ 没有时间衰减 → 两年前的好评代表不了今天
- ❌ 评价者没有身份/声誉 → 互刷无法防范

好的声誉系统 = 身份连续性（Day 6）+ 任务绑定评价 + 结构化 attestation + 经济 stake + 可仲裁 slashing + 可复现 validation。单独某一项都不够。

### 最小实践

为 "DeFi Scout"（Day 6 设计的 Agent）设计一条任务声誉记录：

| 字段 | 内容 |
|------|------|
| 任务 ID | task_defi_scout_001 |
| 任务定义 | 生成 2026-05-27 主流 DeFi 协议收益率简报 |
| 输入 hash | ipfs://QmXxx...（protocol list + 参数） |
| 交付物 hash | ipfs://QmYyy...（markdown 报告） |
| 交付时间 | 2026-05-27 08:00 UTC |
| 付款 | 0.05 ETH，tx: 0xabc... |
| 评价维度 | 准确性 ★★★★☆ / 及时性 ★★★★★ / 可读性 ★★★★☆ |
| 评价者 | 0xUser...（有 3 次历史委托记录） |
| 验证方式 | 人工复查 5 个协议的实际 APY vs 报告数据 |
| 争议窗口 | 7 天，可提交相反证据 |
| 失败条件 | 数据错误 >20% → 退款 100%；未按时交付 → slashing 50% stake |
