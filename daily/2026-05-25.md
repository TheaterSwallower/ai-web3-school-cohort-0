## 学习笔记

### 关键概念

**Machine Payment 核心思想**：不是「AI 会花钱」，而是让机器之间的支付可限制、可验证、可追踪。把「付款意图」和「实际结算」拆开，每一步都有凭证。

**第一性原理三原则**：
1. 预算先于执行 — 没有预算边界就没有安全自动支付
2. 报价必须可比较 — Agent 需要知道价格/币种/有效期/服务范围/退款条件
3. 收据必须可验证 — 付款后能证明付给谁、为什么付、交付了什么

**八大知识节点**：

| 节点 | 难度 | 一句话 |
|------|------|--------|
| Stablecoin Payment | 初级 | 稳定币适合机器支付：价格稳定、可编程、易验证 |
| Budget | 初级 | 多层预算：全局/任务/单次调用/服务方/紧急停止 |
| Quote | 中级 | 可执行报价：价格/币种/收款地址/有效期/quote id |
| Payment Intent | 中级 | 付款意图绑定任务/金额/收款方，不等同于已结算 |
| x402 | 中级 | HTTP 402 原生支付：像处理 401 一样处理付款要求 |
| MPP | 中级 | 机器支付协议：发现→报价→授权→结算→收据 |
| Subscription | 中级 | 订阅模式需要可见性+撤销能力，拒绝静默续费 |
| Micropayment | 高级 | 高频小额：需要 L2/批量结算/链下账本，先算经济账 |

**本章在知识地图的位置**：Machine Payment 是 Agentic Commerce 的基础。前四章解决了「看到什么 → 能做什么 → 怎么决策 → 权限边界」，本章解决 **Agent 怎么花钱、花多少、花给谁**。

### 一条完整支付链路

```
用户授权预算（3 USDC/天）
  → Agent 识别需求、向 API 请求报价
    → API 返回 Quote（0.1 USDC/次，5 分钟有效）
      → Agent 检查：预算够？服务方可信？报价未过期？
        → Wallet/Session Key 执行付款
          → API 返回结果 + Receipt
            → 系统记录：quote / payment intent / tx hash / 结果 / 剩余预算
```

**关键设计点**：
- 预算不是单一数字，要拆成：全局上限 + 任务上限 + 单次上限 + 服务方白名单 + 频率限制
- Quote 必须有有效期和签名，过期不能用
- Approve 和 Pay 分开，approve 是高风险动作必须人工确认
- 小额高频用批量结算，大额单次用 escrow + receipt

### 最小实践

**场景**：Agent 今天最多 3 USDC，需要调用付费数据 API。

| 步骤 | 操作 | 检查点 |
|------|------|--------|
| 1 | 用户授权预算 3 USDC | 写入 wallet policy |
| 2 | API 返回 quote：0.1 USDC/次，5min 有效 | quote id + 签名 |
| 3 | Agent 检查：预算余量 3 ≥ 0.1？服务方在白名单？quote 未过期？ | 三个条件全满足才继续 |
| 4 | 钱包完成 0.1 USDC 付款 | 记录 tx hash |
| 5 | API 返回数据 + receipt | 验证 receipt |
| 6 | 系统更新：剩余 2.9 USDC | 记录完整链路 |

**异常场景**：
- 预算用完 → 拒绝，通知用户追加
- quote 过期 → 重新获取 quote
- 服务方不在白名单 → 人工确认后加入或拒绝
- 付款失败 → 不重试（避免重复扣款），通知用户
