## 学习笔记

### 关键概念

**Agent Wallet 核心思想**：不是「给 AI 一个钱包私钥」，而是把一小部分、可撤销、可审计的权限交给 Agent——让 Agent 可以执行，但只能在用户预先允许的边界内。

**第一性原理**：控制权不能交给一个概率系统。Agent 只能拿到可验证、可限制、可撤销的行动空间。

**三条设计原则**：
| 原则 | 说明 |
|------|------|
| Agent 不直接控制主资产 | 模型可生成计划和交易草稿，但不拿主私钥 |
| 钱包是权限系统 | 表达时间、金额、合约、方法、接收方和撤销条件 |
| 自动化绑定撤销能力 | 用户随时能看到 Agent 权限，并一键关闭 |

**九大知识节点**：
| 节点 | 难度 | 一句话 |
|------|------|--------|
| AA Wallet | 中级 | Account Abstraction 让账户表达规则，不只是私钥控制 |
| Smart Account | 中级 | 合约承载权限/恢复/自动化，是 Agent 的执行边界 |
| Safe | 中级 | 多签 + 模块分散执行权，Agent 参与流程而非独占控制 |
| Session Key | 高级 | 临时受限小钥匙：时间/合约/金额/目标/可撤销 |
| Policy | 高级 | 「Agent 可以做什么」写成系统可检查的规则 |
| Guard | 高级 | 确定性拦截层，拒绝不符合 policy 的交易 |
| Simulation | 中级 | 签名前预演交易结果，翻译成人能看懂的内容 |
| Revocation | 中级 | 权限回收——session key 到期/额度用完/异常行为/用户手动 |
| Human Check | 初级 | 分层确认：低风险自动、中风险模拟后确认、高风险必须确认、超 policy 直接拒绝 |

**本章在知识地图的位置**：Agent Wallet 是 Bridge 的权限骨架。前三章解决了 Agent **看到什么 → 能做什么 → 怎么决策**，本章解决 **Agent 能碰什么、能碰多少、什么时候停**。

### 一个稳的链路

```
用户目标 → Agent 读上下文生成计划 → 系统转受限交易
→ Guard + Simulation 检查风险 → 用户确认关键动作
→ Smart Account / Safe 执行 → 日志记录
```

最容易出问题的是「计划 → 受限交易 → 检查 → 执行」这几步——Demo 很好，没有 Guard/Simulation/Revocation 就不够。

### 最小实践设计

**场景**：Agent 一天最多花 5 USDC，支付白名单 API 服务。

| 要素 | 设计 |
|------|------|
| 额度 | ≤ 5 USDC/天 |
| 时间 | 24h 后自动失效 |
| 目标 | 仅白名单地址（指定 API 服务） |
| 自动执行 | 小额支付（≤ 5 USDC，目标在白名单内） |
| 人工确认 | 任何非白名单目标、任何 approve 操作 |
| 拒绝规则 | 超额度、非白名单地址、session key 过期 |
| 撤销入口 | 用户面板一键 revoke session key |
| 审计记录 | 每笔支付：时间/金额/目标/模拟结果/tx hash |

**四个 case**：
1. 正常：Agent 支付 3 USDC → 自动执行 + 记录
2. 超限：Agent 想付 6 USDC → Policy 拒绝
3. 异常：目标地址不在白名单 → Guard 拦截
4. 用户操作：手动 revoke → Agent 失去执行能力
