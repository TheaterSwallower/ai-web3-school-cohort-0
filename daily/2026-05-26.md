## 学习笔记

### 关键概念

**Agent Identity 核心思想**：不是给 Agent 起名字，而是让用户、服务和其他 Agent 能验证——它是谁、谁控制它、能做什么、入口在哪、历史能不能追溯。

**第一性原理**：Agent 身份 = 控制权绑定 + 能力声明 + 服务入口，缺一不可。

三个必须：
1. **身份要可解析**：别人能从 identifier 找到 profile 和 endpoint
2. **控制权要可证明**：更新 profile 或收款的人要能证明自己是 owner
3. **能力要可验证**：能力声明需要测试、证明、评价或历史记录支撑

**七大知识节点**：

| 节点 | 难度 | 一句话 |
|------|------|--------|
| Agent Profile | 初级 | Agent 公开说明书：名称/描述/能力/价格/接口/钱包/版本 |
| Capability | 中级 | 具体能力声明：输入/输出/价格/限制/风险等级/失败条件 |
| Service Endpoint | 初级 | 外部调用入口：HTTPS/A2A/MCP/Webhook，需认证+限流+版本 |
| Registry | 中级 | 身份登记与发现：链上 registry 提供可验证锚点 |
| DID / VC | 中级 | 去中心化身份 + 可验证声明，跨平台通用身份模型 |
| A2A | 中级 | Agent 间通信协议：发现→协商→委托→返回结果 |
| Ownership | 高级 | 谁控制身份更新和收款：EOA/Smart Account/多签/DAO |

**本章在知识地图的位置**：Agent Identity 是 Agent Trust、Machine Payment 和 Agentic Commerce 的前置条件。前五章解决了「看到 → 能做 → 决策 → 权限 → 支付」，本章解决 **用户/Agent 在和谁打交道**。

### 设计启示

Profile 要同时给人看和给机器读：
- 给人看：描述、owner、收费方式
- 给机器读：endpoint、capabilities、schemas、auth、payment、terms、version

Capability 越具体越好——「我什么都能做」等于什么都没说。需要写清：
- 输入类型 / 输出格式
- 是否需要钱包权限
- 是否会调用外部 API
- 最长执行时间
- 失败如何退款
- 风险等级（只读 / 生成草稿 / 自动执行）

### 最小实践

设计一个 Agent Profile：

| 维度 | 示例 |
|------|------|
| 名称 | "DeFi Scout" |
| 描述 | 监控链上 DeFi 协议变化，生成每日简报 |
| Owner | 0x... 或 DID:example:123 |
| Endpoint | https://api.defiscout.xyz/a2a |
| Capability 1 | 监控收益率变化：输入 protocol list → 输出 APY diff → 0.01 ETH/次 → 低风险 |
| Capability 2 | 生成策略简报：输入 wallet address → 输出 markdown report → 0.05 ETH/次 → 中风险 |
| Capability 3 | 执行小额调仓：输入 strategy params → 输出 tx hash → 0.1 ETH/次 → 高风险（需人工确认） |
| Profile URI | https://defiscout.xyz/.well-known/agent.json 或 ipfs://... |
| 更新机制 | Owner 签名 → 更新 profile → 发布 changelog → 通知订阅用户 |
