## 学习笔记

### 关键概念

**Web3 Tool Use 核心思想**：把 RPC、合约读取、交易生成、钱包确认、区块浏览器和 DeFi 操作变成 Agent 可调用工具——真正难的不是「能调用」，而是权限、参数、模拟和日志。

**第一性原理**：模型可以选择工具，但工具必须用确定性边界限制模型。

**三大铁律**：
| 原则 | 说明 |
|------|------|
| 读写分离 | 读取链上状态和发送交易必须是不同工具、不同权限 |
| 参数结构化 | chain id、contract address、method、args、value、slippage 不能埋在自然语言里 |
| 日志不可省 | 每次工具调用记录输入、输出、时间、来源和错误 |

**八大知识节点**：
| 节点 | 难度 | 一句话 |
|------|------|--------|
| RPC Tool | 初级 | 读链状态（余额/区块/gas），写入能力必须拆出去 |
| Contract Read | 初级 | 调 view/pure 函数，低风险但可能读错网络/ABI |
| Contract Write | 高级 | 写链前需 simulation + policy + 人确认 |
| Wallet Tool | 高级 | 最敏感边界，连接/签名/交易/授权必须分开 |
| Explorer Tool | 初级 | 查询交易/合约/event，提供可验证证据 |
| DeFi Tool | 高级 | swap/借贷需协议白名单+最大额度+滑点限制 |
| Tool Permission | 高级 | 按工具/合约/方法/金额/频率分层授权 |
| Tool Log | 中级 | 每次调用记录：目标→工具→参数→结果→错误→链ID→区块→tx hash→确认人 |

**本章在知识地图的位置**：Web3 Tool Use 是从「AI 能解释链上信息」走向「AI 能参与链上执行」的关键层。越靠近执行，越需要 policy + simulation + human check + 日志。
